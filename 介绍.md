
# 投标文件三项审查系统 — 技术架构

---

一份投标 PDF 从上传到 Excel 出表，走完七个阶段。核心是把三块 RTX 3090 上的视觉大模型和 30 个并行 DeepSeek v4-pro Agent 串成一条流水线。没有正则匹配，没有硬编码模板，任意格式 PDF 通用。

---

## 一、切片上传

Cloudflare 隧道对单个 HTTP 请求有 100MB 的 body 上限。投标文件动辄 300MB 甚至更大，直接传必被拦。

前端 `File.slice(30MB)` 把文件切成几十片。每片单独 POST 到服务端，服务端按索引顺序写入临时文件。全部到齐后执行一次完整文件的 MD5 校验——和客户端上报的哈希比对，确保没有任何一片在传输中损坏。校验通过后才把临时文件合并到最终存储目录，然后通知前端"上传完成"。

断点续传是内置的：前端在任何时候断开，重连后先 GET 一次 `/chunks` 接口查询已成功写入的分片列表，只传缺的那几片。切片上传的全部状态存于内存，会话 24 小时过期自动清理。

---

## 二、PDF 类型判别

不是所有 PDF 都是一样的。判别器用 PyMuPDF 逐页调用 `get_text("text")`，统计每页的文本字符数。一份 700 页的 PDF，0.4 秒扫完。

三个分类阈值：

- 文本覆盖率 ≥ 80% → 纯文本，走 PyMuPDF 原生文本层直读，生成 Markdown，全程不碰 GPU
- 文本覆盖率 ≤ 10% → 纯扫描，必须跑 MinerU hybrid-engine 全量 OCR
- 中间地带 → 混合型，同样走 MinerU hybrid-engine

判别器在分类同时就顺便为纯文本 PDF 构建好了 Markdown：逐页提取文字，按中文章节关键词（一、二、三、投标函、授权委托书、保证金……）自动补 Markdown 标题层级，省掉后续全部 OCR 开销。

---

## 三、MinerU hybrid-engine：双引擎融合的文档解析

这是整套系统的感知层，也是 GPU 算力消耗最大的环节。

hybrid-engine 的设计思路不是"先用 OCR 再过 VLM"，而是让两套本质不同的引擎在统一调度框架下异步交替运行。每一页进入时，调度层根据页面特征实时决策走哪条路径。

**传统 OCR pipeline** 处理文字清晰、版式规整的页面。布局检测模型（PP-DocLayoutV2）先定位文本块和表格边界，OCR 引擎（PP-OCRv4）逐行识别文字，表格结构提取模块输出行列关系。这套管线经过大规模中英文档数据训练，在标准印刷体上精度极高，速度在每页 40-50 毫秒量级——几百页的文字页几分钟扫完。

**VLM 视觉语言模型** 处理复杂页面。印章盖住了关键金额、表格线框残缺、扫描件倾斜、低对比度水印干扰——传统 OCR 的坐标对齐在这些场景下精度大幅下降。VLM 直接把页面渲染为高分辨率图像，从像素层面理解内容。它不是靠坐标算字符位置，而是靠注意力机制对整个页面做全局视觉理解。VLM 天然理解语义，能同时处理文字识别、表格结构、印章遮挡推理——这是传统 OCR 做不到的。

两套引擎异步交替：VLM 推理跑在 GPU 上时，另外两块卡的 OCR 管线照常推进。hybrid-engine 内部有 64 页的处理窗口，窗口内有复杂页就调 VLM，纯文字页走 OCR。三块 RTX 3090 的 72GB 显存池化使用，VLM 模型常驻在卡上不卸载，不存在冷启动延迟。

产出一份完整的结构化 Markdown——带多级标题、`|` 分隔表格、段落结构。同时，MinerU 在解析过程中提取页面级元数据：每段文字来自第几页、每个表格在第几页、每张印章图片在第几页。这些元数据作为 Markdown 注释注入文本流，为后续所有事实提取提供了精确的原始定位锚点。

---

## 四、分片

MinerU 输出的 Markdown 是几十万字符的连续文本流，不能直接喂给 LLM。

分片器按三层匹配逻辑切割：

- 第一层，匹配 Markdown 原生标题——`#`、`##`、`###`
- 第二层，匹配中文投标文档高频章节关键词——二十余个，覆盖投标函、授权委托书、法定代表人证明、投标保证金、项目管理机构组成表、主要人员简历表、劳动合同、社保、企业信誉、信用、营业执照、资质等
- 第三层，兜底均分——前两层都没匹配上时，按每三千字符强制切一段

一份 700 页的投标文件最终产出数百到上千个独立的 Section。每个 Section 作为一个独立的上下文单元，在下一步喂给 LLM Agent。

---

## 五、Map-Reduce：大规模并行事实提取

MinerU 是系统的眼睛，Map-Reduce 是大脑——而且是分布式大脑。

**Map 阶段**：全部 Section 一次性提交给一个 30 线程的并发处理池。每个线程实例化一个 DeepSeek v4-pro Agent，拿到一份独立的 Markdown 片段和一份为投标审查场景专门设计的提取提示词。

这条提示词不是简单的"请提取信息"——它定义了三层提取语义：

第一层，28 类基础事实。按五大类别组织：基础信息（项目名称、招标人、投标人、日期）、报价（投标报价、函报价、附录报价、费率折扣系数、增值税税率）、保证金（声明金额、缴款证明金额、电子保函金额、是否发现各类凭证、保证金形式）、主体信息（法定代表人、授权代理人、统一社会信用代码、注册地址、开户银行、银行账号）、响应信息（投标有效期、服务期限）。每个字段都有精确的提取定义和证据要求。

第二层，人员社保。姓名、拟任职务、职称、证书名称及编号、专业、社保状态、劳动合同关联。每个人员记录必须对应原文中真实存在的自然人，严禁编造。角色限定为项目管理团队的标准构成——项目负责人、各类专业负责人、其他成员——法定代表人和授权代理人已在主体信息层处理，不在此列。

第三层，表格结构化。自动识别 Markdown 中表格的类型（人员表、报价表、保证金表、社保表），输出行列数、置信度评分和前几行内容预览，为后续交叉校验提供结构化参照。

每条事实附带：原文摘录（可定位回原文的 30 字片段）、置信度评分（0-1）、来源页码。这三者的组合构成完整的证据回溯链。

**Reduce 阶段**：所有 Agent 的返回汇聚上来。同一个 `field_name` 可能出现在多个章节——"投标人名称"在投标函、授权书、保证金凭证中至少各出现一次。Reduce 引擎按字段名分组，取最高置信度那条作为主结果，合并所有原文摘录聚合为完整证据链。人员按姓名加证书编号的组合键去重——同一个人出现在两页里不会算两条。

最后跑一次全局一致性 LLM：把所有已提取的事实和人员作为输入，让 DeepSeek 从全局视角做跨章节交叉校验——报价金额是否一致、保证金凭证是否匹配、人员证书有无冲突。输出 findings，作为 Excel 的"判断建议" Sheet。

---

## 六、异步处理与渐进式进度

审查管线是长耗时任务——OCR 2-4 分钟，LLM 提取 2 分钟，加上校验和出表，单次审查 4-6 分钟。用传统的 HTTP 长连接（SSE 或 WebSocket）会触发 Cloudflare 100 秒超时，导致前端报 524 连接断开。

我们用的是异步轮询模式：审查启动接口 50 毫秒返回 `runId`，后端在独立线程中跑完整管线。前端每 500 毫秒 GET 一次 `/status` 接口查询进度——每次请求都是毫秒级短连接，永远不触发 CF 超时。

进度不是估算的。MinerU OCR 阶段按实际处理的页面数推进度条；LLM 提取阶段每完成 10 段汇报一次进度；每一步耗时精确计时。前端看到的不是"加载中……"，而是"文档解析 287 秒 / LLM 并行提取 180/330 (02:34)"。

---

## 七、七表 Excel 输出

最终产物是固定结构的七 Sheet Excel：概览、基础事实提取、无法提取原因、判断建议、人员社保提取、判断依据范围、需补充材料清单。

每条事实行携带：PDF 实际页码、原文摘录、置信度评分、分析备注。敏感字段——身份证号、银行卡号、社保编号——在渲染层做掩码脱敏，只输出不可逆的脱敏版本。风险项红色标注，需人工复核项明确列出补充材料建议。
